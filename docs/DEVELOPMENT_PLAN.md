## 상세 작업 항목

### Phase 1: 프로젝트 기반 구축

#### 1.1 백엔드 프로젝트 초기화 (Go)
- **배경:** CDL 플랫폼의 모든 API를 제공할 Go 백엔드 서버를 구축해야 한다.
- **작업 내용:**
  - Go 프로젝트 구조 생성 (cmd/, internal/, pkg/)
  - HTTP 라우터 설정 (chi 또는 gin)
  - 설정 관리 (환경변수, config 파일)
  - 로깅 및 미들웨어 기본 설정
  - Docker 개발 환경 구성
- **인수 조건:**
  - `go build`가 성공한다
  - Health check 엔드포인트(`GET /api/health`)가 200을 반환한다
  - Docker Compose로 로컬 개발 환경이 구동된다
- **의존성:** 없음 (최초 작업)

#### 1.2 데이터베이스 스키마 설계 및 마이그레이션
- **배경:** 챌린지, 수강생, 진행 상황, 평가 결과 등의 데이터를 저장할 DB가 필요하다.
- **작업 내용:**
  - 데이터베이스 선정 (PostgreSQL 권장)
  - ERD 설계: users, challenges, realworld_apps, enrollments, challenge_steps, evaluations 테이블
  - 마이그레이션 도구 설정 (golang-migrate 또는 goose)
  - 초기 마이그레이션 파일 작성
  - 시드 데이터 스크립트 작성
- **인수 조건:**
  - 마이그레이션이 정상 실행되어 모든 테이블이 생성된다
  - 시드 데이터가 정상 삽입된다
  - 마이그레이션 롤백이 정상 동작한다
- **의존성:** 1.1 (백엔드 프로젝트 초기화)

#### 1.3 프론트엔드 프로젝트 구조 정리
- **배경:** 현재 목업으로 구현된 프론트엔드를 실제 개발에 맞게 구조를 정리해야 한다.
- **작업 내용:**
  - API 클라이언트 레이어 구축 (fetch wrapper)
  - 공통 컴포넌트 분리 (Badge, Card, Table, Modal, ProgressBar 등)
  - 상태 관리 방식 결정 및 설정
  - 환경변수 설정 (API 엔드포인트 등)
  - 목 데이터를 API 호출로 교체할 수 있는 구조 마련
- **인수 조건:**
  - 공통 컴포넌트가 분리되어 재사용 가능하다
  - API 클라이언트가 설정되어 백엔드 엔드포인트를 호출할 수 있다
  - 빌드가 에러 없이 통과한다
- **의존성:** 없음 (목업 기반으로 독립 진행 가능)

### Phase 2: 인증 시스템

#### 2.1 회원가입/로그인 API 및 UI 연동
- **배경:** MVP에서 유일한 인증 기능으로, 수강생과 관리자가 시스템에 접근하기 위해 필수적이다.
- **작업 내용:**
  - 회원가입 API (`POST /api/auth/signup`)
  - 로그인 API (`POST /api/auth/login`)
  - JWT 토큰 발급 및 검증 미들웨어
  - 사용자 역할 관리 (student, admin)
  - 프론트엔드 로그인/회원가입 폼 API 연동
  - 인증 상태 관리 (토큰 저장, 자동 로그아웃)
  - 보호된 라우트 구현 (인증 필요 페이지 접근 제어)
- **인수 조건:**
  - 이메일/비밀번호로 회원가입이 가능하다
  - 로그인 시 JWT 토큰이 발급된다
  - 인증되지 않은 사용자는 보호된 페이지에 접근할 수 없다
  - 관리자/수강생 역할이 구분된다
- **의존성:** 1.1, 1.2, 1.3

---

### Phase 3: 챌린지 관리 (관리자)

#### 3.1 챌린지 CRUD API
- **배경:** 관리자가 챌린지를 등록하고 관리할 수 있어야 한다.
- **작업 내용:**
  - 챌린지 생성 API (`POST /api/admin/challenges`)
  - 챌린지 목록 조회 API (`GET /api/challenges`)
  - 챌린지 상세 조회 API (`GET /api/challenges/:id`)
  - 챌린지 수정 API (`PUT /api/admin/challenges/:id`)
  - 챌린지 삭제 API (`DELETE /api/admin/challenges/:id`)
  - 관리자 권한 검증 미들웨어
- **인수 조건:**
  - 관리자가 챌린지를 생성/조회/수정/삭제할 수 있다
  - 비관리자는 관리 API에 접근할 수 없다 (403)
  - 챌린지에 제목, 설명, 기술스택, 난이도, 5단계 구조가 포함된다
- **의존성:** 2.1 (인증 시스템)

#### 3.2 리얼월드 앱 연결 관리
- **배경:** 챌린지는 리얼월드 앱 구현체를 기반으로 진행되므로, 앱과 챌린지를 연결 관리해야 한다.
- **작업 내용:**
  - 리얼월드 앱 CRUD API
  - 챌린지-리얼월드 앱 연결 API
  - GitHub 리포 URL 검증
- **인수 조건:**
  - 리얼월드 앱을 등록하고 챌린지와 연결할 수 있다
  - 앱 정보에 이름, 기술스택, GitHub 리포 URL이 포함된다
- **의존성:** 3.1 (챌린지 CRUD API)

#### 3.3 관리자 챌린지 관리 UI 연동
- **배경:** 목업으로 구현된 관리자 챌린지 관리 페이지를 실제 API와 연동해야 한다.
- **작업 내용:**
  - 챌린지 목록 테이블 API 연동
  - 챌린지 등록 모달 API 연동
  - 챌린지 수정/삭제 기능 연동
  - 리얼월드 앱 선택 UI 구현
  - 에러 처리 및 로딩 상태 표시
- **인수 조건:**
  - 관리자 페이지에서 챌린지 등록/수정/삭제가 실제 동작한다
  - 리얼월드 앱을 선택하여 챌린지에 연결할 수 있다
  - 로딩 및 에러 상태가 적절히 표시된다
- **의존성:** 3.1, 3.2, 1.3

---

### Phase 4: 챌린지 수강 (수강생)

#### 4.1 챌린지 목록/상세 API 및 UI 연동
- **배경:** 수강생이 챌린지를 탐색하고 상세 정보를 확인할 수 있어야 한다.
- **작업 내용:**
  - 챌린지 목록 조회 API (필터링, 페이지네이션)
  - 기술스택별 필터링 기능
  - 챌린지 상세 조회 API
  - 프론트엔드 목록/상세 페이지 API 연동
- **인수 조건:**
  - 수강생이 기술스택별로 챌린지를 필터링할 수 있다
  - 챌린지 상세 페이지에서 5단계 구조와 리얼월드 앱 정보를 확인할 수 있다
- **의존성:** 3.1 (챌린지 CRUD API), 1.3

#### 4.2 챌린지 등록 및 진행 상태 관리
- **배경:** 수강생이 챌린지에 등록하고, 단계별 진행 상태를 추적해야 한다.
- **작업 내용:**
  - 챌린지 등록 API (`POST /api/enrollments`)
  - 진행 상태 업데이트 API (`PUT /api/enrollments/:id/progress`)
  - 수강생별 등록 챌린지 조회 API
  - GitHub 리포 포크/생성 연동 (webhook 또는 GitHub API)
  - 진행 상태 UI 반영 (스텝퍼 업데이트)
- **인수 조건:**
  - 수강생이 챌린지에 등록할 수 있다
  - 단계별 진행 상태가 저장되고 UI에 반영된다
  - 챌린지 시작 시 GitHub 리포가 생성/포크된다
- **의존성:** 4.1, 2.1

#### 4.3 수강생 대시보드 API 및 UI 연동
- **배경:** 수강생이 자신의 전체 학습 현황을 한눈에 파악할 수 있어야 한다.
- **작업 내용:**
  - 수강생 대시보드 데이터 집계 API
  - 진행 중/완료 챌린지 현황 API
  - 프론트엔드 대시보드 API 연동
- **인수 조건:**
  - 대시보드에 진행 중/완료 챌린지 수와 평균 점수가 표시된다
  - 각 챌린지의 진행률이 정확히 반영된다
  - 완료된 챌린지의 평가 결과가 표시된다
- **의존성:** 4.2 (챌린지 등록 및 진행 상태)

### Phase 5: 티칭모드 플러그인

#### 5.1 티칭모드 스킬 프레임워크 설계
- **배경:** Claude Code가 티칭모드로 동작하려면 스킬(.md) 형태의 플러그인 프레임워크가 필요하다.
- **작업 내용:**
  - 티칭모드 스킬 .md 파일 구조 설계
  - 챌린지 단계별 가이드 프롬프트 설계 (결과물 정의→문서화→테스트→훅→스킬)
  - 스킬 설치/활성화 메커니즘 설계
  - 스킬과 웹 플랫폼 간 진행 상태 동기화 방안 설계
- **인수 조건:**
  - 티칭모드 스킬 .md 파일 스펙이 문서화된다
  - 샘플 스킬 파일이 Claude Code에서 정상 로드된다
  - 결과물 정의에 집중하도록 가이드하는 프롬프트가 검증된다
- **의존성:** 없음 (독립적으로 설계 가능)

#### 5.2 챌린지별 티칭모드 스킬 템플릿 구현
- **배경:** 각 챌린지(기술스택)에 맞는 맞춤형 티칭모드 스킬이 필요하다.
- **작업 내용:**
  - 챌린지별 스킬 템플릿 생성 시스템
  - 기본 제공 챌린지 6종에 대한 스킬 파일 작성
  - 스킬 파일 다운로드/설치 API
  - 웹 플랫폼에서 스킬 설치 가이드 UI
- **인수 조건:**
  - 6개 기본 챌린지에 대한 티칭모드 스킬이 작성된다
  - 수강생이 웹에서 설치 명령어를 복사하여 Claude Code에 설치할 수 있다
  - 티칭모드에서 단계별 가이드가 정상 동작한다
- **의존성:** 5.1, 3.1

### Phase 6: 자동 평가 시스템

#### 6.1 CI/CD 파이프라인 (GitHub Actions) 구축
- **배경:** 수강생 산출물의 품질을 자동으로 검증하는 CI/CD 파이프라인이 필요하다.
- **작업 내용:**
  - 챌린지별 GitHub Actions 워크플로우 템플릿 작성
  - 테스트 실행, 린트, 커버리지 체크 자동화
  - 평가 결과를 웹훅으로 CDL 서버에 전송하는 메커니즘
  - 챌린지 리포 생성 시 워크플로우 자동 포함
- **인수 조건:**
  - 수강생이 코드를 push하면 자동으로 테스트가 실행된다
  - 테스트 통과/실패 결과가 CDL 서버로 전달된다
  - 기술스택별 워크플로우 템플릿이 존재한다
- **의존성:** 4.2 (챌린지 등록 - GitHub 리포 생성)

#### 6.2 Claude Code 리뷰 연동
- **배경:** CI/CD 외에 Claude Code가 산출물의 품질을 검토하고 피드백을 제공해야 한다.
- **작업 내용:**
  - Claude API 연동 (산출물 품질 평가)
  - 평가 기준 프롬프트 설계 (문서화 품질, 테스트 커버리지, 훅/스킬 구현 품질)
  - 평가 결과 저장 API
  - 피드백 생성 및 점수 산출 로직
- **인수 조건:**
  - 수강생 산출물에 대해 Claude Code가 품질 점수(0-100)를 산출한다
  - 구체적인 개선 피드백이 생성된다
  - 평가 결과가 DB에 저장된다
- **의존성:** 6.1 (CI/CD 결과 수신 후 트리거)

#### 6.3 평가 결과 웹 플랫폼 연동
- **배경:** 자동 평가 결과를 수강생과 관리자가 웹에서 확인할 수 있어야 한다.
- **작업 내용:**
  - 평가 결과 조회 API
  - 수강생 챌린지 상세 페이지에 평가 결과 표시
  - 수강생 대시보드에 최근 평가 결과 표시
  - 관리자 모니터링 페이지에 평가 결과 통합
- **인수 조건:**
  - 수강생이 챌린지 상세 페이지에서 CI/CD 결과와 Claude Code 피드백을 확인할 수 있다
  - 관리자가 전체 평가 결과를 모니터링할 수 있다
- **의존성:** 6.2, 4.3, 7.1

### Phase 7: 관리자 모니터링

#### 7.1 수강생 모니터링 API 및 UI 연동
- **배경:** 한빛미디어 담당자가 수강생들의 학습 현황을 실시간으로 파악할 수 있어야 한다.
- **작업 내용:**
  - 수강생 목록 및 진행 상황 조회 API
  - 수강생별 상세 진행 정보 API
  - 프론트엔드 모니터링 페이지 API 연동
  - 수강생 행 확장/축소 상세 뷰 연동
- **인수 조건:**
  - 관리자가 전체 수강생 목록과 진행 상황을 조회할 수 있다
  - 수강생별 챌린지 진행 단계, CI/CD 상태, Claude Code 점수를 확인할 수 있다
- **의존성:** 4.2, 2.1

#### 7.2 통계 대시보드
- **배경:** 전체적인 교육 효과를 파악하기 위한 통계가 필요하다.
- **작업 내용:**
  - 전체 통계 집계 API (수강생 수, 완료율, 평균 점수 등)
  - 챌린지별 통계 (등록 수, 완료 수, 평균 완료 시간)
  - 통계 대시보드 UI 구현
- **인수 조건:**
  - 관리자 대시보드에 전체 수강생 수, 완료율, 평균 점수가 표시된다
  - 챌린지별 통계를 확인할 수 있다
- **의존성:** 7.1

### Phase 8: 배포 및 운영

#### 8.1 AWS 인프라 구축 (IaC)
- **배경:** CDL 플랫폼을 AWS에 안정적으로 배포하기 위한 인프라가 필요하다.
- **작업 내용:**
  - Terraform 또는 AWS CDK로 인프라 코드 작성
  - VPC, ECS/EKS, RDS, S3, CloudFront 구성
  - 환경 분리 (staging, production)
  - 모니터링 및 알림 설정 (CloudWatch)
- **인수 조건:**
  - IaC 코드로 인프라가 재현 가능하게 배포된다
  - staging/production 환경이 분리되어 있다
  - 기본 모니터링 및 알림이 설정되어 있다
- **의존성:** 없음 (인프라는 독립적으로 준비 가능)

#### 8.2 CI/CD 배포 파이프라인
- **배경:** 코드 변경이 자동으로 테스트되고 배포되는 파이프라인이 필요하다.
- **작업 내용:**
  - GitHub Actions 배포 워크플로우 (백엔드, 프론트엔드)
  - 자동 테스트 → 빌드 → 배포 파이프라인
  - 환경별 배포 전략 (staging 자동, production 수동 승인)
  - 롤백 절차 문서화
- **인수 조건:**
  - main 브랜치 push 시 staging에 자동 배포된다
  - production 배포는 수동 승인 후 진행된다
  - 롤백이 1분 내에 가능하다
- **의존성:** 8.1 (AWS 인프라)
